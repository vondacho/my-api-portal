# CLAUDE.md

Guidance for Claude Code (and any contributor) working on this platform.

## Project Overview
This repository is part of an API contract lifecycle platform that parses, validates, scores, and applies overlays to API contracts (OpenAPI, AsyncAPI, GraphQL SDL, WSDL), and serves content through Strapi as a headless CMS. The platform is made up of:

- **Onboarding module** (Java/SpringBoot3/Strapi) — REST microservice that exposes core business logic, contract parsing, validation, scoring, and overlaying. Persistence through Strapi integration, scoring through Scoring module integration.
- **Scoring module** (Node.js/Typescript/Spectral) — REST microservice that exposes linting and scoring logic through Spectral integration.
- **Catalogue module** (Astro.js) — Web frontend that provides listing, search, and detail viewing of specification resources. Onboarding module integration.
- **Registry module** (Strapi) — REST microservice, Headless CMS, treated strictly as an external system reached via its REST API.

## Architectural Principles
- Hexagonal Architecture (Ports & Adapters).
- Contract-First API Design.
- Gang-of-Four pattern.

## Component Guidance
### Catalogue module
- Types are generated from the contract-first OpenAPI spec via `openapi-typescript` — don't hand-write types that duplicate the contract.
- Treat the spec as upstream and read-only from the frontend's perspective; spec changes happen in the backend/contract pipeline, not in frontend code.

### Strapi Integration
- Strapi is always reached over its REST API — never imported as a library or accessed directly from domain/use-case code.
- All Strapi-specific concerns (auth, pagination, response shape, retries, fallbacks) live in the adapter behind the outbound port, not in the domain.

## Testing
- RestAssured for End-to-end testing of the onboarding REST API.
- Microcks mocking engine for mocking Strapi REST API.
- Microcks for contract testing of the onboarding REST API.
- Cucumber for functional testing of the registration use case.
- JUnit 5 for testing the domain layer of the Onboarding module.

## Before Writing Code
- If a task would require importing a framework/library type into the domain or port layer, stop and surface that an adapter-layer abstraction is missing — don't leak the dependency to make progress.
- If a task touches resilience (retry/circuit breaker/timeout) logic, confirm it's being added to an adapter, not a use case or domain service.
- When generating clients or types from a spec, regenerate from the contract — don't hand-edit generated code.
